# Autovacuum Timeout and ProcArrayLock Contention Analysis

## Observed Symptoms

```
2026-01-07 14:06:32 UTC::@:[1464] autovacuum worker took too long to start; canceled
2026-01-07 14:06:36 UTC::@:[1464] autovacuum worker took too long to start; canceled
2026-01-07 14:07:03 UTC::@:[53776] autovacuum worker started without a worker entry
```

Accompanied by:
- Sudden spike in active sessions (thousands)
- Heavy `LWLock:ProcArray` contention (yellow in AAS graph)
- `IPC:ProcArrayGroupUpdate` waits (grey in AAS graph)
- CPU saturation (green in AAS graph)

## Root Cause Analysis

### The Autovacuum Startup Race Condition

The two warning messages indicate a **race condition** in autovacuum worker startup:

1. **Launcher requests worker** - Sets `av_startingWorker` and records `wi_launchtime` (`src/backend/postmaster/autovacuum.c:1279`)

2. **Timeout calculation** - The launcher waits up to `Min(autovacuum_naptime, 60)` seconds (`src/backend/postmaster/autovacuum.c:678`):
   ```c
   waittime = Min(autovacuum_naptime, 60) * 1000;
   if (TimestampDifferenceExceeds(worker->wi_launchtime, current_time, waittime))
   ```

3. **Timeout triggers** - If the worker doesn't clear `av_startingWorker` in time, the launcher cancels the entry and logs "took too long to start; canceled" (`src/backend/postmaster/autovacuum.c:702-703`)

4. **Orphan worker** - The slow worker eventually starts, finds `av_startingWorker = NULL`, and logs "started without a worker entry" (`src/backend/postmaster/autovacuum.c:1563`)

### Why Workers Are Slow to Start: ProcArrayLock Contention

Every new backend (including autovacuum workers) must call `ProcArrayAdd()` during initialization, which requires **exclusive** locks:

```c
// src/backend/storage/ipc/procarray.c:478-479
LWLockAcquire(ProcArrayLock, LW_EXCLUSIVE);
LWLockAcquire(XidGenLock, LW_EXCLUSIVE);
```

This conflicts with:

| Operation | Lock Mode | Code Location |
|-----------|-----------|---------------|
| `ProcArrayAdd` (new connections) | EXCLUSIVE | `procarray.c:478` |
| `ProcArrayGroupClearXid` (commits) | EXCLUSIVE | `procarray.c:849` |
| `GetSnapshotData` (statement start) | SHARED | `procarray.c:2181` |
| `ComputeXidHorizons` (vacuum) | SHARED | `procarray.c:1695` |

### The Cascade Effect

1. **High transaction commit rate** triggers many backends needing `ProcArrayLock` in EXCLUSIVE mode

2. **Group commit optimization** (`ProcArrayGroupClearXid`) causes followers to wait on `IPC:ProcArrayGroupUpdate` (the grey bars in the graph):
   ```c
   // src/backend/storage/ipc/procarray.c:829
   pgstat_report_wait_start(WAIT_EVENT_PROCARRAY_GROUP_UPDATE);
   ```

3. **New connections queue** - All new backends (including autovacuum workers) block waiting for EXCLUSIVE lock

4. **O(n) operations under lock** - `ProcArrayAdd` performs memmove operations proportional to session count:
   ```c
   // src/backend/storage/ipc/procarray.c:515-526
   memmove(&arrayP->pgprocnos[index + 1], ...);
   memmove(&ProcGlobal->xids[index + 1], ...);
   memmove(&ProcGlobal->subxidStates[index + 1], ...);
   memmove(&ProcGlobal->statusFlags[index + 1], ...);
   ```

5. **Vicious cycle** - More sessions = longer lock hold time = more queueing = autovacuum timeouts

## The NULL Wait Event Spikes

"NULL" wait event indicates either:
1. **Active CPU usage** (not waiting on any resource)
2. **Uncovered/unreported wait event** (code path missing wait event instrumentation)

The spikes could represent:
- Backends actively processing (not blocked)
- Lock acquisition spinning in code paths without wait event reporting
- The O(n) memmove operations in `ProcArrayAdd`
- Commit group processing
- **Potentially uninstrumented waits** in the autovacuum or lock manager code paths

## Recommendations

### Immediate Investigation

1. **Identify the trigger at ~14:06:35**
   - Connection pool reset?
   - Application deployment/restart?
   - Bulk operation spawning many transactions?
   - Periodic batch job?

2. **Check connection metrics**
   ```sql
   SELECT count(*) FROM pg_stat_activity;
   -- Compare max_connections vs actual at spike time
   ```

3. **Monitor ProcArray lock contention**
   ```sql
   SELECT wait_event, count(*)
   FROM pg_stat_activity
   WHERE wait_event_type = 'LWLock'
   GROUP BY wait_event;
   ```

### Configuration Considerations

1. **Connection pooling** - Reduce direct connections to minimize `ProcArrayAdd` calls

2. **`autovacuum_naptime`** - This directly affects the timeout (capped at 60s)

3. **`max_connections`** - Lower values reduce O(n) impact in `ProcArrayAdd`

### Long-term

The fundamental issue is that `ProcArrayLock` is a global contention point. High-throughput workloads with many connections will always stress this lock. Consider:
- Connection pooling (PgBouncer, Odyssey) to reduce backend churn
- Workload analysis to reduce transaction commit frequency
- Application-level connection management improvements
