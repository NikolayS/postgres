# Autovacuum Timeout and ProcArrayLock Contention Analysis

**PostgreSQL Version: 16.8 (REL_16_8)**

## Observed Symptoms

```
2026-01-07 14:06:32 UTC::@:[1464] autovacuum worker took too long to start; canceled
2026-01-07 14:06:36 UTC::@:[1464] autovacuum worker took too long to start; canceled
2026-01-07 14:07:03 UTC::@:[53776] autovacuum worker started without a worker entry
```

Accompanied by:
- Sudden spike in active sessions (thousands)
- Heavy `LWLock:ProcArray` contention (yellow in AAS graph)
- `IPC:ProcArrayGroupUpdate` waits (grey in AAS graph)
- CPU saturation (green in AAS graph)

## Root Cause Analysis

### The Autovacuum Startup Race Condition

The two warning messages indicate a **race condition** in autovacuum worker startup:

1. **Launcher requests worker** - Sets `av_startingWorker` and records `wi_launchtime` (`src/backend/postmaster/autovacuum.c:1329-1331`)

2. **Timeout calculation** - The launcher waits up to `Min(autovacuum_naptime, 60)` seconds (`src/backend/postmaster/autovacuum.c:740-741`):
   ```c
   waittime = Min(autovacuum_naptime, 60) * 1000;
   if (TimestampDifferenceExceeds(worker->wi_launchtime, current_time, waittime))
   ```

3. **Timeout triggers** - If the worker doesn't clear `av_startingWorker` in time, the launcher cancels the entry and logs "took too long to start; canceled" (`src/backend/postmaster/autovacuum.c:763-765`)

4. **Orphan worker** - The slow worker eventually starts, finds `av_startingWorker = NULL`, and logs "started without a worker entry" (`src/backend/postmaster/autovacuum.c:1685`)

### Why Workers Are Slow to Start: ProcArrayLock Contention

Every new backend (including autovacuum workers) must call `ProcArrayAdd()` during initialization (`src/backend/storage/lmgr/proc.c:474-481`), which requires **exclusive** locks:

```c
// src/backend/storage/ipc/procarray.c:479-480
LWLockAcquire(ProcArrayLock, LW_EXCLUSIVE);
LWLockAcquire(XidGenLock, LW_EXCLUSIVE);
```

This conflicts with:

| Operation | Lock Mode | Code Location |
|-----------|-----------|---------------|
| `ProcArrayAdd` (new connections) | EXCLUSIVE | `procarray.c:479` |
| `ProcArrayGroupClearXid` (commits) | EXCLUSIVE | `procarray.c:849` |
| `GetSnapshotData` (statement start) | SHARED | `procarray.c:2257` |
| `ComputeXidHorizons` (vacuum) | SHARED | `procarray.c:1741` |

### The Cascade Effect

1. **High transaction commit rate** triggers many backends needing `ProcArrayLock` in EXCLUSIVE mode

2. **Group commit optimization** (`ProcArrayGroupClearXid` at line 795) causes followers to wait on `IPC:ProcArrayGroupUpdate` (the grey bars in the graph):
   ```c
   // src/backend/storage/ipc/procarray.c:829
   pgstat_report_wait_start(WAIT_EVENT_PROCARRAY_GROUP_UPDATE);
   ```

3. **New connections queue** - All new backends (including autovacuum workers) block waiting for EXCLUSIVE lock

4. **O(n) operations under lock** - `ProcArrayAdd` performs memmove operations proportional to session count:
   ```c
   // src/backend/storage/ipc/procarray.c:515-527
   memmove(&arrayP->pgprocnos[index + 1], ...);
   memmove(&ProcGlobal->xids[index + 1], ...);
   memmove(&ProcGlobal->subxidStates[index + 1], ...);
   memmove(&ProcGlobal->statusFlags[index + 1], ...);
   ```

5. **Vicious cycle** - More sessions = longer lock hold time = more queueing = autovacuum timeouts

### Timeline Reconstruction

```
14:06:25-14:06:35  Normal operation
~14:06:35          Triggering event (connection storm / commit storm)
14:06:32           First autovacuum worker requested
14:06:32           Worker blocked on ProcArrayLock (can't complete ProcArrayAdd)
14:06:32 + 60s     Timeout fires → "took too long to start; canceled"
14:06:36           Second attempt, same result
14:06:40+          Cascade in full effect - session count explodes
14:07:03           Orphan worker finally acquires lock, finds no entry
```

## The NULL Wait Event Spikes

"NULL" wait event indicates either:
1. **Active CPU usage** (not waiting on any resource)
2. **Uncovered/unreported wait event** (code path missing wait event instrumentation)

The spikes could represent:
- Backends actively processing (not blocked)
- Lock acquisition spinning in code paths without wait event reporting
- The O(n) memmove operations in `ProcArrayAdd`
- Commit group processing
- **Potentially uninstrumented waits** in the autovacuum or lock manager code paths

## Recommendations

### Immediate Investigation

1. **Identify the trigger at ~14:06:35**
   - Connection pool reset?
   - Application deployment/restart?
   - Bulk operation spawning many transactions?
   - Periodic batch job?

2. **Check connection metrics**
   ```sql
   SELECT count(*) FROM pg_stat_activity;
   -- Compare max_connections vs actual at spike time
   ```

3. **Monitor ProcArray lock contention**
   ```sql
   SELECT wait_event, count(*)
   FROM pg_stat_activity
   WHERE wait_event_type = 'LWLock'
   GROUP BY wait_event;
   ```

### Configuration Considerations

1. **Connection pooling** - Reduce direct connections to minimize `ProcArrayAdd` calls

2. **`autovacuum_naptime`** - This directly affects the timeout (capped at 60s)

3. **`max_connections`** - Lower values reduce O(n) impact in `ProcArrayAdd`

### Long-term

The fundamental issue is that `ProcArrayLock` is a global contention point. High-throughput workloads with many connections will always stress this lock. Consider:
- Connection pooling (PgBouncer, Odyssey) to reduce backend churn
- Workload analysis to reduce transaction commit frequency
- Application-level connection management improvements

## Related Changes in Subsequent Versions

### PostgreSQL 16.9

No changes directly related to this issue. Only minor fix:
- `2f33de3cdbc` - Fix rare assertion failure in standby replay

### PostgreSQL 17

**Backend ID simplification** (`ab355e3a88d`):
- Redefined backend ID to be an index into the proc array
- Simplifies some internal bookkeeping but does **not** fundamentally change ProcArrayLock contention

**Wait event description fix** (`18e3e290bc6`, backported to 16.x):
- Fixed misleading descriptions for `ProcArrayGroupUpdate` and `XactGroupUpdate`
- Changed "end of a parallel operation" to "transaction end" (these events are about commit, not parallel queries)

### PostgreSQL 18 (Development)

Several relevant improvements:

**`c758119e5bf` - Allow changing autovacuum_max_workers without restarting**
- Introduces new parameter `autovacuum_worker_slots` (requires restart)
- `autovacuum_max_workers` can now be changed with SIGHUP
- Allows dynamic adjustment during contention events

**`306dc520b9d` - Introduce autovacuum_vacuum_max_threshold**
- New parameter to cap the vacuum trigger calculation
- Default: 100,000,000 tuples
- Helps ensure very large tables are vacuumed more frequently
- Can reduce overall vacuum-related lock pressure

**`6897f0ec024` - Optimize memory access in GetRunningTransactionData()**
- Minor optimization: performs cheap XID comparison before fetching database ID
- Reduces cache misses under ProcArrayLock

### Core Issue Status

**The fundamental ProcArrayLock scalability limitation has NOT been addressed** in any released version. The lock remains a global contention point for:
- New backend startup (`ProcArrayAdd`)
- Transaction commit (`ProcArrayGroupClearXid`)
- Snapshot acquisition (`GetSnapshotData`)

The 60-second autovacuum worker startup timeout also remains unchanged. For high-connection-count workloads, connection pooling remains the primary mitigation strategy.
